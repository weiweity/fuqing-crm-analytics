## 1. 产品概述

### 1.1 产品名称
芙清 CRM 客户分析系统 (Fuqing CRM Analytics)

### 1.2 产品定位
内部运营中台工具，为芙清电商运营团队提供数据驱动的客户洞察能力，支撑日常运营决策和月度复盘。

### 1.3 核心价值主张
- **每日 9 点自动推送**：运营团队每天上班即获得最新客户洞察
- **智能问数**：通过自然语言查询，快速获取数据答案
- **一键导出复盘**：自动生成 PPT/飞书文档，支撑月度复盘会议
- **口径唯一可信**：所有指标统一注册、统一文档、统一维护，前后端 100% 对齐

### 1.4 目标用户

| 用户角色 | 使用频率 | 核心场景 | 成功标准 |
|---------|---------|---------|---------|
| 运营专员（你） | 每天 | 查看数据、发现问题、生成洞察 | 5 分钟内获得可行动的洞察 |
| 运营经理 | 每天 | 掌握团队整体情况、分配资源 | 3 分钟了解全局 |
| 老板/创始人 | 每周/每月 | 听汇报、参与复盘 | 10 分钟理解业务现状 |
| 开发/AI Agent | 按需 | 新增指标、修复口径、前端对接 | 10 分钟内找到指标定义和 API 契约 |

## 2. 核心问题与痛点

### 2.1 现状痛点
1. **无数据驱动**：产品迭代凭感觉，不知道为什么爆、为什么好
2. **Excel 太慢**：做完分析洞察已过时，无法支撑实时决策
3. **文件太大**：2GB 数据无法追溯历史，做不了长期分析
4. **机会唯一**：全公司只有你能做，是升职带团队的关键项目
5. **口径混乱**（v2.0 新增）：同一个 GMV 前后端理解不一致，改一个定义要改十几个文件

### 2.2 成功指标
- **效率提升**：数据获取时间从 2 小时降至 5 分钟
- **洞察时效**：从 T+7 提升至 T+1（次日 9 点）
- **决策支撑**：每周至少产生 1 个可执行的运营建议
- **口径一致性**：核心指标（GSV/GMV/新老客）定义只存在于 1 个位置，改一处全局生效

## 3. 功能需求

### 3.1 Week 1: 核心指标看板

#### 3.1.1 数据清洗与标准化
- **输入**: 25-26 年店铺数据库 + 会员数据库（约 50 个 xlsx 文件）
- **输出**: 清洗后的 DuckDB 数据库
- **清洗规则**:
  - 统一时间格式（下单时间、付款时间）
  - 标准化商品标题（提取 SKU、规格、品类）
  - 用户 ID 去重与关联
  - 金额字段统一（元为单位，处理退款）
  - **新增（v2.0）**：所有有效订单判定规则统一引用语义层 `OrderFilters`

#### 3.1.2 核心指标模块

| 模块 | 指标 | 计算逻辑 | 展示方式 | 语义层定义位置 |
|------|------|---------|---------|---------------|
| **GMV 概览** | 总 GMV | 实付金额汇总 | 卡片 + 趋势图 | `semantic/metrics.py` → `gmv` |
| | 订单数 | 去重订单编号计数 | 卡片 + 趋势图 | `semantic/metrics.py` → `order_count` |
| | 客单价 | GMV / 订单数 | 卡片 | `semantic/metrics.py` → `avg_order_value` |
| **GSV 概览** | 总 GSV | 实付金额（剔除退款+购物金） | 卡片 + 趋势图 | `semantic/metrics.py` → `gsv` |
| | GSV 订单数 | 有效订单计数 | 卡片 | `semantic/metrics.py` → `gsv_order_count` |
| **新老客分析** | 新客数 | 窗口内首次购买用户数 | 卡片 + 饼图 | `semantic/metrics.py` → `new_users` |
| | 老客数 | 窗口前已有购买记录的用户数 | 卡片 + 饼图 | `semantic/metrics.py` → `old_users` |
| | 新客 GMV | 新客实付金额 | 卡片 | `semantic/metrics.py` → `new_gsv` |
| | 老客 GMV | 老客实付金额 | 卡片 | `semantic/metrics.py` → `old_gsv` |
| **会员分析** | 会员数 | 有会员标识的用户数 | 卡片 | `semantic/metrics.py` → `member_users` |
| | 会员 GMV | 会员实付金额 | 卡片 | `semantic/metrics.py` → `member_gsv` |
| | 会员占比 | 会员 GMV / 总 GMV | 百分比 | `semantic/metrics.py` → `member_gsv_ratio` |
| **同环比** | 同比 | 当月 vs 去年同期 | 增长率 + 箭头 | `semantic/time.py` |
| | 环比 | 当月 vs 上月 | 增长率 + 箭头 | `semantic/time.py` |

#### 3.1.3 筛选能力
- 时间范围：日/周/月/自定义
- 店铺：芙清旗舰店（默认，后续可扩展）
- 产品维度：全部/单品类/单 SKU
- **人群维度**：channel / spu_tier / spu_product_class / spu_product_subclass（4 种，v2.0 统一由 `semantic/dimensions.py` 注册管理）

## 5. 数据定义

### 5.1 有效订单（v2.0 统一定义）

所有业务统计中的"有效订单"必须引用同一来源：`backend/semantic/filters.py`。

当前定义：
```sql
order_status LIKE '%成功%'
AND is_goujinjin = FALSE
AND is_refund = FALSE
```

### 5.2 老客定义（月维度滚动窗口）

以 2026-02-25 为例：
- 老客 = 在 2026-02-01 之前有过购买记录的用户
- 新客 = 在 2026-02-01 至 2026-02-25 首次购买的用户

**v2.0 新增**：首购日期统一来源于 `user_first_purchase` 表，cutoff = 分析窗口起始日期 - 1 天。

### 5.3 会员定义
- 有会员标识的用户（根据数据中的 `is_member` 字段判断， LEFT JOIN 会员数据库）

### 5.4 GMV 定义
- GMV = 实付金额（`actual_amount`），不剔除退款和购物金

### 5.5 GSV 定义（v2.0 新增）
- GSV = 实付金额（`actual_amount`），仅统计有效订单（剔除退款和购物金）
- 定义位置：`backend/semantic/metrics.py` → `gsv`

### 5.6 渠道 9 层漏斗（v2.0 新增）

定义位置：`backend/semantic/channels.py`

| 层级 | 名称 | 判定规则摘要 |
|------|------|-------------|
| P1 | u先派样 | 关键词匹配 |
| P2 | 百补派样 | 关键词匹配 |
| P3 | 赠品&0.01 | 关键词/金额匹配 |
| P4 | 达播/微博 | 关键词/id 直接覆盖匹配 |
| P5 | 淘客 | 淘客商品 ID 列表命中 |
| P6 | 购物金 | 购物金关键词匹配 |
| P7 | 直播 | 直播数据源 CSV 命中 |
| P8 | 货架 | 未命中以上任何规则的订单 |
| P9 | 其他 | 无法识别的渠道 |

## 8. 附录

### 8.1 术语表
- **GMV**: Gross Merchandise Value，商品交易总额
- **GSV**: Gross Sales Value，有效销售额（剔除退款和购物金）
- **RFM**: Recency（最近购买）、Frequency（频率）、Monetary（金额）
- **SKU**: Stock Keeping Unit，库存单位
- **语义层 (Semantic Layer)**: 统一封装业务口径、指标公式、维度定义的技术中间层
- **API 契约 (API Contract)**: 前后端共同遵守的接口数据结构约定，通过 Pydantic + OpenAPI 文档化

## 9. 语义层与口径管理架构（v2.0 新增）

### 9.1 架构目标

解决以下 4 个已确认的问题：
1. **口径碎片化**：同一个过滤条件写在 6+ 个文件里
2. **规则硬编码**：8象限定义在 4 个 service 里各写一份
3. **契约混乱**：Pydantic 模型内联在 `main.py`，前端猜字段
4. **无指标注册中心**：新增指标全靠口口相传

### 9.2 架构设计

采用 SaaS 公司标准的 **语义层 + 契约层 + 服务层** 三层架构：

```
前端 (Vue3) ←── openapi-typescript ──→ API 契约层 (Pydantic)
                      ↑
                服务层 (services) — 只负责编排查询
                      ↑
         ┌─────────────────────────────┐
         │      语义层 (semantic)       │
         │  filters  → 统一 SQL 过滤条件 │
         │  metrics  → 指标注册表        │
         │  dimensions → 维度注册表      │
         │  segments → 8象限/RFM 定义    │
         │  channels → 9层渠道漏斗       │
         │  time     → MTD/YoY/MoM 构造  │
         └─────────────────────────────┘
                      ↑
                数据层 (DuckDB)
```

### 9.3 模块说明

| 模块 | 文件路径 | 职责 | 禁止事项 |
|------|----------|------|----------|
| 过滤器 | `backend/semantic/filters.py` | 生成统一 SQL WHERE 条件 | 禁止在 Service 中硬编码 `order_status LIKE '%成功%'` |
| 指标注册表 | `backend/semantic/metrics.py` | 注册所有指标的名称、公式、口径 | 禁止在 Service 中内联写 `SUM(actual_amount)` |
| 维度注册表 | `backend/semantic/dimensions.py` | 注册所有分析维度 | 禁止在 Service 中硬编码 GROUP BY 表达式 |
| 人群分层 | `backend/semantic/segments.py` | 8象限定义 + RFM 阈值 + SQL 生成 | 禁止在多个 service 中重复写 CASE WHEN |
| 渠道定义 | `backend/semantic/channels.py` | 9层渠道漏斗规则文档化 | 禁止前后端对渠道名称理解不一致 |
| 时间构造器 | `backend/semantic/time.py` | MTD/自由模式/MoM/YoY 日期计算 | 禁止在 Service 中重复写日期逻辑 |
| API 契约 | `backend/contracts/schemas.py` | 所有 Pydantic Request/Response 模型 | 禁止在 `main.py` 中内联定义 Response 模型 |

### 9.4 新增/修改指标 SOP

**Step 1**: 在 `backend/semantic/metrics.py` 注册指标
**Step 2**: 在 `backend/contracts/schemas.py` 更新 Response 模型字段
**Step 3**: 在 Service 中通过 `MetricRegistry.get("xxx").sql_expr` 引用
**Step 4**: 后端启动后，前端执行 `npx openapi-typescript ...` 自动生成 TS 类型
**Step 5**: 前端使用自动生成的类型调用 API

### 9.5 关键原则

- **单一真实来源**：任何口径只定义一次
- **改一处全局生效**：`filters.py` 中的有效订单定义变更后，所有引用它的 service 自动生效
- **契约优先**：前后端开发前必须先确认 `contracts/schemas.py` 中的模型定义
- **类型自动生成**：前端禁止手写 guess API 返回类型
