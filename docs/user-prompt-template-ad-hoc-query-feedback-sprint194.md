# Sprint 194 运营话术模板预读反馈

预读方式: mock 预读。Codex 按业务组运营同事视角走读 `docs/user-prompt-template-ad-hoc-query.md`; 由于本地实施阶段没有可联系的真实业务同事姓名或通讯渠道, 本反馈需在 Stage 3 后由 Claude/业务侧补一次真人复核。

预读日期: 2026-07-02

## 1. 可读性

结论: 可直接复制使用, 对非技术运营同事友好。

- 模板 1/2/3 都把日期范围、按天、指标、必用 tool 写在同一句里, 能绕过 WorkBuddy 决策层误判。
- 第 3 节关键词表足够短, 运营同事能快速定位"小样/派样/会员/新客/老客 + 按天 + 多周期"必须走 `daily-gsv-multi-period`。
- 第 4 节"工具缺位自检"表达清楚, 能减少把已有 tool 误判为缺位的情况。

非阻塞建议: 模板 4 已说明全店单维度可用 `daily_gsv` 或 `yoy_battle`, 与第 1 节强提示不冲突; 若后续业务反馈仍误读, 可在 v1.1 增加"只有涉及小样/派样/会员/新老客按天多周期时才强制 daily-gsv-multi-period"。

## 2. 覆盖度

结论: 覆盖 Sprint 183/193 触发过的核心场景。

- 已覆盖小样/派样、会员、新客、老客、8 维度、2 周期/多周期、全店 GSV、两年 30 指标、不确定路由等高频问法。
- 已把 `daily-gsv-multi-period` 放在模板、关键词表、自检步骤三处重复强调, 能降低漏用 tool 的风险。
- "只有 3 种情况才是真工具缺位"的边界足够实用: 超过 8 维度、需要明细行、指标不在 8 个 enum 内。

非阻塞建议: 可在后续 v1.1 明确举例说明 AUS/客单价属于"指标不在 8 个 enum 内"的自检边界, 防止运营把客单价也误认为 daily-gsv-multi-period 必然支持。

## 3. 实战

结论: 按运营常见口语走读, 能稳定落到正确 tool。

- "帮我看小样和会员这两段时间每日变化" -> 命中小样/会员 + 按天 + 两周期, 应使用 `daily-gsv-multi-period`。
- "新客老客按天对比去年同期, 顺便给 GSV" -> 命中新老客 + 按天 + 周期对比, 应使用 `daily-gsv-multi-period`。
- "只看今年 618 全店 GSV 同比" -> 属于全店单维度, 可走 `daily_gsv` 或 `yoy_battle`, 不应报工具缺位。

综合判断: 模板 v1.0 可进入运营试用; 本次 mock 预读未发现必须立即修改话术模板的阻塞问题。
