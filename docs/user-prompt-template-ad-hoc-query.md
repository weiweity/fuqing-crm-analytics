# 芙清 CRM 运营问数话术模板 (WorkBuddy 专用)

## 1. 必用 daily-gsv-multi-period tool 强提示

> Sprint 193 拍板: 提到「小样 / 派样 / 会员 / 新客 / 老客 + 按天 + 多周期 / 2 周期」时, 必用 daily-gsv-multi-period tool. WorkBuddy 不要走 `daily_gsv` (全店) 或 `new_old_customer` (汇总) 凑数, 不要报"工具缺位", 不要写临时脚本.

## 2. 高频 5 个 prompt 模板 (复制即用)

### 模板 1: 多年同时间段对比 (派样 / 会员 / 新老客按天)

> 帮我跑: 2026-05-06 至 2026-06-21 跟 2025-05-06 至 2025-06-21 两个周期, 按天输出小样 GMV/GSV、会员 GMV/GSV、新客人数/GSV、老客人数/GSV 8 个指标. 必用 daily-gsv-multi-period tool.

### 模板 2: 单周期小样 8 维度按天

> 帮我跑: 2026-06-01 至 2026-06-30, 按天输出 8 个维度 (小样 / 会员 / 新老客 x GMV/GSV/人数). 必用 daily-gsv-multi-period tool.

### 模板 3: 单维度多周期 (只 GMV)

> 帮我跑: 2026-05-06 至 2026-06-21 跟 2025-05-06 至 2025-06-21 两个周期, 按天输出小样 GMV 跟会员 GMV 两个指标. 必用 daily-gsv-multi-period tool.

### 模板 4: 大促 YOY 战斗

> 帮我跑: 2026-06-01 至 2026-06-18 跟 2025-06-01 至 2025-06-18 两个周期, 按天输出全店 GSV 同比. 这个只要全店单维度, 可用 `daily_gsv` 或 `yoy_battle`.

### 模板 5: 不确定要哪个 tool

> 帮我用 `ask` 路由查一下: "小样 + 会员 + 按天 + 多周期对比", 看返回什么 tool + args; 如果返回 daily-gsv-multi-period 就直接跑.

## 3. 关键词必查表

| 运营词 | 必用 tool |
|---|---|
| 小样 / 派样 + 按天 + 多周期 | `daily-gsv-multi-period` |
| 会员 + 按天 + 多周期 | `daily-gsv-multi-period` |
| 新老客 + 按天 + 多周期 | `daily-gsv-multi-period` |
| 8 维度 / 8 指标 | `daily-gsv-multi-period` |
| 周期对比 / 2 周期 / 两个窗口 | `daily-gsv-multi-period` |
| 全店 GSV 趋势 (单周期) | `daily_gsv` |
| 两年 30 指标对比 | `two_year_overview` |
| 不确定 | `ask` |

## 4. 报"工具缺位"自检 4 步

1. 查本模板第 3 节关键词必查表.
2. 查 `~/.claude/skills/ad-hoc-query/SKILL.md` 第 1.5.1 节关键词同义词库.
3. 调 `ask(query)` 关键词路由.
4. 只有 3 种情况才是真工具缺位: 超过 8 维度、需要明细行、指标不在 8 个 enum 内. 其他大多数运营问数都已有现成 tool.

## 5. Sprint 194 mock 预读反馈 (2026-07-02)

> **预读方式**: Codex 按业务组运营同事视角走读本模板. 由于本地实施阶段没有可联系的真实业务同事姓名或通讯渠道, 本反馈需在 Stage 3 后由 Claude/业务侧补一次真人复核. 标 fix_pattern #80 "mock 预读待真人复核".

### 5.1 可读性

**结论**: 可直接复制使用, 对非技术运营同事友好.

- 模板 1/2/3 都把日期范围、按天、指标、必用 tool 写在同一句里, 能绕过 WorkBuddy 决策层误判.
- 第 3 节关键词表足够短, 运营同事能快速定位"小样/派样/会员/新客/老客 + 按天 + 多周期"必须走 `daily-gsv-multi-period`.
- 第 4 节"工具缺位自检"表达清楚, 能减少把已有 tool 误判为缺位的情况.

**非阻塞建议**: 模板 4 已说明全店单维度可用 `daily_gsv` 或 `yoy_battle`, 与第 1 节强提示不冲突; 若后续业务反馈仍误读, 可在 v1.1 增加"只有涉及小样/派样/会员/新老客按天多周期时才强制 daily-gsv-multi-period".

### 5.2 覆盖度

**结论**: 覆盖 Sprint 183/193 触发过的核心场景.

- 已覆盖小样/派样、会员、新客、老客、8 维度、2 周期/多周期、全店 GSV、两年 30 指标、不确定路由等高频问法.
- 已把 `daily-gsv-multi-period` 放在模板、关键词表、自检步骤三处重复强调, 能降低漏用 tool 的风险.
- "只有 3 种情况才是真工具缺位"的边界足够实用: 超过 8 维度、需要明细行、指标不在 8 个 enum 内.

**非阻塞建议**: 可在后续 v1.1 明确举例说明 AUS/客单价属于"指标不在 8 个 enum 内"的自检边界, 防止运营把客单价也误认为 daily-gsv-multi-period 必然支持.

### 5.3 实战走读

**结论**: 按运营常见口语走读, 能稳定落到正确 tool.

- "帮我看小样和会员这两段时间每日变化" -> 命中小样/会员 + 按天 + 两周期, 应使用 `daily-gsv-multi-period`.
- "新客老客按天对比去年同期, 顺便给 GSV" -> 命中新老客 + 按天 + 周期对比, 应使用 `daily-gsv-multi-period`.
- "只看今年 618 全店 GSV 同比" -> 属于全店单维度, 可走 `daily_gsv` 或 `yoy_battle`, 不应报工具缺位.

**综合判断**: 模板 v1.0 可进入运营试用; 本次 mock 预读未发现必须立即修改话术模板的阻塞问题. **真人复核待业务组运营同事 5-10 分钟预读** (Sprint 195 R4 续).
