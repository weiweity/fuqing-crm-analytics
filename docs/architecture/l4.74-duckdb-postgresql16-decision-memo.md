## 2. 选型对比 (Alternatives)

### 2.1 PostgreSQL 16 分布式 (本次 L4.74 推荐)

| 维度 | 评估 |
|---|---|
| **类型** | 关系型 OLTP + 列存扩展 (citus 分布式 / pg_partman 分区) |
| **数据规模** | 单节点 100GB+ 性能强, citus 横向扩展 shard/replica (跟 ClickHouse 1:1 stable 沿用) |
| **SQL 兼容** | **100% 兼容 DuckDB SQL** (PostgreSQL 是 SQL 标准, DuckDB 99% 兼容 PostgreSQL 1:1 stable 沿用) |
| **学习曲线** | 低, 业务方都用过 PostgreSQL, 不需要培训 |
| **运维** | 单节点用 docker-compose, citus cluster 需协调节点 + worker 节点 |
| **生态** | Citus (分布式) / PgBouncer (连接池) / pg_partman (时间分区) / pgcrypto (加密) |
| **大厂对比** | 字节跳动 / 美团 / 拼多多 OLTP+OLAP 双用, 跟 DuckDB 兼容最好 |
| **ROI** | **高**, 适合 "现有 DuckDB 95%+ SQL 直接迁移 + OLTP 事务支持" 场景 |
| **成本** | 单节点 (32C64G + 1TB NVMe) ~ 0.8 万/年 (开源免费); 3 节点 citus cluster ~ 2.4 万/年 |

**L4.74 推荐理由**:
1. **DuckDB 兼容性 100%**: PostgreSQL 16 SQL 100% 兼容 DuckDB (DuckDB 99% 兼容 PostgreSQL, 跟 L4.51 Read-Write Splitting 1:1 stable 沿用)
2. **OLTP + OLAP 双用**: 业务方既有取数需求 (OLAP) 又有事务需求 (OLTP, 跟 L4.51 1:1 stable 沿用)
3. **运维简单**: 单机部署跟现有 DuckDB 一样, 业务方无感
4. **成本最低**: 开源免费, 跟 ClickHouse Cloud (1 万/年) 和 Trino cluster (5 万/年) 比 0 成本
5. **替代风险最小**: 现有 DuckDB SQL 直接迁移, 改造量 5-10% (跟 Trino 1:1 stable 沿用)

### 2.2 ClickHouse (跟 L4.56 1:1 stable 沿用, 备选)

| 维度 | 评估 |
|---|---|
| **类型** | 列存 OLAP (column-oriented) |
| **数据规模** | 单节点 100GB+ 性能强, 横向扩展 shard/replica |
| **SQL 兼容** | 90% 兼容 DuckDB/PostgreSQL 语法, 但 DuckDB 特有 (`SELECT * EXCLUDE` / `LIST` / `STRUCT`) 需要重写 |
| **学习曲线** | 低, SQL 方言跟 PostgreSQL 类似 |
| **运维** | 需要专门 DBA, 单节点可用 docker-compose, cluster 需 zookeeper |
| **生态** | ClickHouse Cloud (托管) / Altinity (K8s operator) |
| **大厂对比** | 字节跳动 / 腾讯 / 快手 OLAP 主力, DataWorks 内置 |
| **ROI** | 中, 适合"查询层列存 + ETL 输出 Parquet" 场景 |
| **成本** | 单节点 (32C64G + 1TB NVMe) ~ 1 万/年; 3 节点 cluster ~ 3 万/年 |

**ClickHouse 备选场景** (未来): 如果业务是 "纯列存 OLAP 查询" (无 OLTP 事务需求), 选 ClickHouse 列存性能更优.

### 2.3 Trino (原 PrestoSQL) (跟 L4.56 1:1 stable 沿用, 备选)

| 维度 | 评估 |
|---|---|
| **类型** | 分布式 SQL 查询引擎 (federated query) |
| **数据规模** | 不存数据, 需要外部存储 (S3/HDFS/Hive/Iceberg), 查询层分布式 |
| **SQL 兼容** | 95% 兼容 ANSI SQL, 跨数据源联邦查询 (MySQL/PG/Hive/S3 同 query) |
| **学习曲线** | 中, 需要懂 connector 架构 + 资源组配置 |
| **运维** | 需要专门 DBA, coordinator + worker 集群, resource groups 限制并发/内存/CPU/扫描量 |
| **生态** | Starburst (托管) / Ahana (K8s operator) |
| **大厂对比** | Meta / Uber / 字节跳动 联邦查询主力, DataWorks 内置 |
| **ROI** | 高, 适合"查询层分布式 + 不动数据层" 场景 |
| **成本** | coordinator (8C16G) + worker (3 × 16C32G) ~ 5 万/年 + S3 存储费 |

**Trino 备选场景** (未来): 如果业务是 "跨数据源联邦查询" (MySQL/PG/Hive/S3 都在 Trino 联邦查询), 选 Trino.

### 2.4 选型推荐: PostgreSQL 16 分布式 (跟 L4.74 启动条件 c 真触发 1:1 stable 永久规则链配套)

**理由**:

2. **OLTP + OLAP 双用**: 业务方既有取数需求 (OLAP) 又有事务需求 (OLTP), 跟 L4.51 Read-Write Splitting 1:1 stable 沿用
3. **支持 10+ 业务分析师并发取数**: 启动条件 c 真触发 (10 > 5 阈值), 跟 L4.69 RFM 雪崩 8 并发 1:1 stable 沿用, 跟 L4.72.2 dual_conn semaphore timeout 治本 配套
4. **查询 P95 < 5s 期望**: 跟 Sprint 202+ R6 跑批 wall_min 估算 14-18min 1:1 stable 沿用
5. **运维可接受**: 单机部署跟现有 DuckDB 一样, 业务方无感
6. **成本最低**: 开源免费, 跟 ClickHouse Cloud (1 万/年) 和 Trino cluster (5 万/年) 比 0 成本
7. **替代风险最小**: 现有 DuckDB SQL 直接迁移, 改造量 5-10% (跟 Trino 1:1 stable 沿用)

**ClickHouse 备选场景** (未来): 如果业务是 "纯列存 OLAP 查询" (无 OLTP 事务需求), 选 ClickHouse 列存性能更优.
