# 50M 行规模架构方案

**文档版本**: v1.0
**创建日期**: 2026-06-09
**依据**: 50M 行 benchmark 数据 (2026-06-09 跑批实测, 见 `data/processed/fuqing_crm_50m.duckdb` 7.3 GB)

---

## 1. 当前架构

### 1.1 现状

| 指标 | 值 |
|------|-----|
| **orders 表行数** | ~10.6M |
| **DuckDB 文件大小** | 67.8 GB（含预计算表 + 索引） |
| **DuckDB 版本** | 1.5.3 |
| **后端** | FastAPI + Uvicorn（单进程） |
| **前端** | Vue 3 |
| **部署** | 本地 macOS / 单机 |

### 1.2 现有瓶颈

- `fact_rfm_long` 预计算表膨胀文件体积（67.8GB vs 原始数据约 7GB）
- 单进程后端，无并发查询能力
- 无数据分区，全表扫描场景多

## 2. Benchmark 结果摘要

### 2.1 查询性能（单查询，3 次平均）

| 场景 | 10.6M | 50M | 倍率 |
|------|-------|-----|------|
| 全店复购率 | 0.129s | 0.556s | 4.31x |
| RFM 分群（实时） | 0.624s | 6.772s | 10.86x |
| 渠道占比 | 0.321s | 1.066s | 3.32x |
| 30 指标对比 | 0.760s | 2.280s | 3.00x |
| **合计** | **1.835s** | **10.674s** | **5.82x** |

> 数据量增长 ~4.7x，查询耗时增长 ~5.8x，接近线性扩展。

### 2.2 内存使用

| 指标 | 10.6M | 50M |
|------|-------|-----|
| RSS 峰值 | 2,336 MB | 3,913 MB |
| 16GB 占比 | 14.6% | 24.5% |
| 安全余量 | 充足 | 充足 |

### 2.3 文件大小

| 库 | 大小 | 说明 |
|----|------|------|
| 10.6M 库 | 67.8 GB | 含预计算表 + 索引，严重膨胀 |
| 50M 库（仅 orders） | 7.3 GB | 列式压缩效果极佳 |

### 2.4 并发性能

| 数据库 | 1 Worker | 4 Workers | 退化倍率 |
|--------|----------|-----------|----------|
| 10.6M | 689ms | 2.31s | 3.35x |
| 50M | 4.19s | 22.17s | 5.29x |

> 4 workers 并发时 groupby_channel 出现 11.56x 退化，但后端为单进程，实际无影响。

## 4. 架构方案选项

### 方案 A：继续 DuckDB 单文件（推荐）

**描述**: 保持现有 DuckDB 单文件架构，补充预计算表 + 索引优化。

**优势**:
- 零迁移成本，现有代码无需改动
- DuckDB 列式压缩 50M 行仅 7.3GB，磁盘不是瓶颈
- 单进程后端天然避免并发竞争问题
- Benchmark 已验证可行性

**劣势**:
- 单文件上限约 100-200M 行（受内存限制）
- 无法水平扩展
- 数据增长到 100M+ 时需重新评估

**优化措施**:
1. 为 50M 库补充 `fact_rfm_long` 预计算表（P0）
2. 添加 `pay_time` 索引加速时间范围查询（P1）
3. 定期 `ANALYZE` 更新统计信息（P2）
4. 调整 `memory_limit` 至 4GB（P1）

**预期效果**: RFM 查询从 6.8s 降至 ~0.002s，综合看板加载 <2s。

---

### 方案 B：DuckDB 分区表

**描述**: 按年/月对 orders 表进行分区，减少单分区数据量。

**优势**:
- 时间范围查询只扫描相关分区
- 单分区数据量降至 ~5-10M 行
- 对现有代码改动较小（SQL 层面适配）

**劣势**:
- DuckDB 分区支持有限（需手动管理分区文件）
- 增加 ETL 复杂度（分区维护、跨分区查询）
- 跨年度查询需 UNION ALL，增加代码复杂度
- 50M 行规模下收益不明显（benchmark 已验证性能可接受）

**适用场景**: 数据量超过 100M 行，且查询多为时间范围限定。

---

### 方案 C：迁移到 PostgreSQL

**描述**: 将数据从 DuckDB 迁移到 PostgreSQL，利用其成熟的并发和扩展能力。

**优势**:
- 成熟的并发处理能力（多连接、连接池）
- 丰富的索引类型（B-tree、GIN、GiST）
- 支持物化视图（预计算）
- 社区生态完善，长期维护成本低

**劣势**:
- 迁移成本高：需重写所有 SQL 查询（DuckDB 方言 vs PG 方言）
- 部署复杂度增加：需维护 PG 服务 + 连接池
- 列式存储性能不如 DuckDB（行存储 vs 列存储）
- 50M 行规模下 PG 的优势不明显，DuckDB 已足够
- 单机部署场景下 PG 的并发优势无法体现

**迁移成本评估**:
- SQL 适配：~2-3 天（方言差异、函数替换）
- ETL 改造：~1-2 天（COPY 语句、批量插入）
- 测试验证：~1-2 天
- 总计：~5-7 天

**适用场景**: 需要多用户并发访问、或数据量超过 200M 行。

---

### 方案 D：混合方案（热数据 DuckDB + 冷数据 Parquet）

**描述**: 近 1-2 年数据存 DuckDB（热数据），历史数据存 Parquet 文件（冷数据），通过 UNION ALL 或视图统一查询。

**优势**:
- 热数据查询保持 DuckDB 高性能
- 冷数据压缩率更高（Parquet 列式存储）
- 可按需加载冷数据，降低内存占用
- DuckDB 原生支持读取 Parquet 文件

**劣势**:
- 增加 ETL 复杂度（数据分层、生命周期管理）
- 跨层查询需 UNION ALL，增加代码复杂度
- 冷数据查询性能取决于 Parquet 文件大小和分区策略
- 50M 行规模下收益不明显

**适用场景**: 数据量超过 100M 行，且有明确的冷热分层需求。

---

## 5. 推荐方案

### 推荐：方案 A（DuckDB 单文件 + 优化）

**理由**:

1. **Benchmark 已验证可行性**: 50M 行下查询耗时 3-6 倍线性增长，最慢场景 ~6.8s（无预计算），补充预计算表后 <2s，完全可接受。

2. **零迁移成本**: 现有代码、ETL 流程、部署架构无需改动，节省 5-7 天迁移成本。

3. **资源充足**: 文件大小 7.3GB（压缩后），内存峰值 3.9GB，16GB 机器完全够用。

4. **单进程后端天然适配**: Uvicorn 单进程避免了 DuckDB 的并发竞争问题，benchmark 中 4 workers 的 11.56x 退化在生产环境不会出现。

5. **未来扩展路径清晰**: 若数据量增长到 100M+，可平滑升级到方案 B 或 D，无需推翻现有架构。

**不推荐其他方案的原因**:

| 方案 | 不推荐原因 |
|------|-----------|
| B（分区） | 50M 行规模下收益不明显，增加 ETL 复杂度 |
| C（PostgreSQL） | 迁移成本 5-7 天，50M 行规模下 DuckDB 性能已足够 |
| D（混合） | 50M 行规模下冷热分层收益不明显，增加复杂度 |

## 6. 实施计划

### Phase 1：预计算表补充（P0，1 天）

| 步骤 | 内容 | 耗时 |
|------|------|------|
| 1.1 | 为 50M 库生成 `fact_rfm_long` 预计算表 | 2h |
| 1.2 | 验证 RFM 查询性能（目标 <0.01s） | 0.5h |
| 1.3 | 更新 ETL 脚本，50M 库纳入日常刷新 | 1h |
| 1.4 | 跑全量 pytest 验证 | 0.5h |

### Phase 2：索引和统计优化（P1，0.5 天）

| 步骤 | 内容 | 耗时 |
|------|------|------|
| 2.1 | 为 orders 表添加 `pay_time` 索引 | 0.5h |
| 2.2 | 运行 `ANALYZE` 更新统计信息 | 0.5h |
| 2.3 | 验证查询计划（EXPLAIN） | 0.5h |

## 7. 风险和应对

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 预计算表生成耗时过长 | 低 | 中 | 分批生成，或降级为实时计算 |
| 查询性能不达预期 | 低 | 高 | 补充索引 + ANALYZE，或降级到 10.6M 库 |
| 内存不足 | 低 | 高 | 调整 memory_limit，或升级机器 |
| DuckDB 版本兼容性 | 低 | 低 | 锁定 1.5.3 版本 |

---

## 8. 未来扩展路径

当数据量增长到 100M+ 行时，按以下路径平滑升级：

```
当前（50M，方案 A）
    ↓ 数据量 100M+
方案 B（分区表）
    ↓ 数据量 200M+ 或需要多用户并发
方案 C（PostgreSQL）或 方案 D（混合）
```

每次升级都有明确的触发条件和迁移路径，避免一次性大规模重构。
