# 品类流转Tab增量PRD

## 1. 产品目标

优化品类看板的"品类流转"tab，提升运营团队分析品类间用户流转路径的效率，支持更长周期（365天）的追溯，全量展示流转矩阵，并提供百分比视角快速发现核心流转路径。

## 2. 变更摘要（按优先级）

### P0 - 必须完成

| # | 变更项 | 说明 |
|---|--------|------|
| 1 | 新增365天追溯窗口 | 时间窗口选择器增加"365天"选项，后端SQL支持365天流转计算 |
| 2 | 渠道排除：赠品&0.01、其他 | 这两个渠道的订单不参与品类流转分析（SQL过滤层排除） |
| 3 | 流转矩阵全量展示 | 不限于TOP10，返回全部品类；前端优先展示TOP10，支持滚轮下滑浏览全部 |
| 4 | 矩阵百分比/数值切换 | 默认百分比显示（行百分比），点击tab切换为数值 |
| 5 | 表头冻结 | 横向滚动时表头固定，纵向滚动时首列固定 |

### P1 - 重要优化

| # | 变更项 | 说明 |

| 7 | 百分比计算逻辑 | 行百分比 = 从A流向B的人数 / 从A流出的总人数 |

## 3. 数据口径定义

### 3.1 流转分析口径

- **分析周期**: start_date ~ end_date（用户选择的日期范围）
- **追溯窗口**: 在分析周期基础上向前扩展 window_days 天，即 (start_date - window_days) ~ end_date
- **首购定义**: 在追溯窗口内，用户按时间排序的第一单所属品类
- **次购定义**: 在追溯窗口内，用户按时间排序的第二单所属品类
- **流转定义**: 首购品类A → 次购品类B，记为一条从A到B的流转

### 3.2 渠道排除口径

- **默认排除**: '赠品&0.01'、'其他' 两个渠道的订单不参与品类流转分析
- **排除位置**: SQL WHERE 子句层面排除，确保这些订单不进入首购/次购计算
- **与现有逻辑关系**: 独立于 `excludeLowPrice` 筛选，是品类流转tab的固有逻辑

### 3.3 百分比口径

- **行百分比**: 矩阵中 (A→B) 的百分比 = 从A流向B的人数 / 从A流出的总人数 × 100%
- **数值**: 矩阵中直接展示流转人数（原始值）
- **默认值**: 百分比模式

## 4. 需求详情

### 4.1 时间窗口选择器

**变更**: 增加 "365天" 选项

```
原选项: [30天, 90天, 180天]
新选项: [30天, 90天, 180天, 365天]
```

**影响**: 
- 前端 `WINDOW_OPTIONS` 数组增加一项
- 后端 `window_days` 参数已支持任意数值，无需改造

### 4.2 渠道排除

**变更**: 品类流转分析默认排除 '赠品&0.01' 和 '其他' 渠道

**实现方式**:
- 前端：无需新增UI开关，在调用API时默认传入 `exclude_channels=['赠品&0.01', '其他']`
- 后端：SQL中通过 `o.channel NOT IN (?, ?)` 排除

**注意**: 此排除仅影响品类流转分析的数据计算，不影响其他tab

### 4.3 流转矩阵全量展示

**当前问题**:
- 矩阵被限制为10x10（top_n=10）
- 非TOP10品类被归类为"其他"，丢失了详细流转信息

**新设计**:
- 后端返回全部品类的流转矩阵（不再限制top_n）
- 前端表格默认展示TOP10来源品类 × TOP10目标品类
- 用户可通过鼠标滚轮下滑查看全部来源品类
- 不做分页，采用固定高度 + 内部滚动条

**前端交互**:
- 表格高度固定（如500px）
- 垂直方向：滚动查看所有来源品类
- 水平方向：滚动查看所有目标品类
- 表头冻结（Naive UI NDataTable 支持 `fixed: 'left'` 和 `fixed: 'right'`）

### 4.4 百分比/数值切换

**交互设计**:
- 矩阵卡片标题栏增加一个切换按钮组：[百分比 | 数值]
- 默认选中"百分比"
- 切换时表格数据实时更新，不重新请求API

**百分比计算**:
```
percentage(A→B) = flow(A→B) / sum(flow(A→*)) × 100%
```

**显示格式**:
- 百分比：保留1位小数，如 "23.5%"
- 数值：整数，千分位分隔，如 "1,234"
- 0值显示为 "—"

### 4.5 前后置关联分析

**当前实现**:
- 已存在 pre_purchase（之前买了啥）和 post_purchase（之后买了啥）两个表格
- 数据口径：分析周期内购买过目标品类的用户，在window_days内购买的其他品类

**优化点**:
- 保持现有上下排列的布局
- 表格增加更清晰的标题说明
- 数据口径保持不变（已合理）

## 5. API变更

### 5.1 请求参数变更

```typescript
// 新增：流向方向参数（用于矩阵百分比计算）
interface CategoryFlowParams {
  start_date: string
  end_date: string
  level?: string
  top_n?: number        // 改为可选，不传时返回全部
  window_days?: number
  channel?: string
  exclude_channels?: string[]  // 默认 ['赠品&0.01', '其他']
  target_category?: string
}
```

### 5.2 响应数据变更

```typescript
interface FlowMatrix {
  sources: string[]        // 全部来源品类（按流转量降序）
  targets: string[]        // 全部目标品类（按流转量降序）
  matrix: number[][]       // 全量矩阵（数值）
  row_totals: number[]     // 每行总和（用于前端计算百分比）
  concentration_warnings: string[]
}
```

**关键变更**:
- `matrix` 返回全量数据，不再限制10x10
- 新增 `row_totals` 字段，前端用于计算行百分比
- `sources` 和 `targets` 按流转量降序排列

## 6. 待确认问题

1. **全量矩阵性能**: 如果品类数量很多（如50+），矩阵可能达到50x50=2500个单元格，前端渲染性能是否可接受？
   - 建议：Naive UI NDataTable 的虚拟滚动可解决

2. **渠道排除的灵活性**: 是否需要在UI上提供开关，让用户可以选择不排除这两个渠道？
   - 建议：P0阶段不加开关，默认排除；如后续有需求再增加

3. **365天窗口的首购定义**: 365天窗口内，用户的首购定义是否仍然合理？（某些用户可能在365天前就有首购）
   - 当前口径：在追溯窗口内的第一单为首购，这是合理的

## 7. 非目标

1. 不改桑基图的展示逻辑（保持现有TOP10+其他的节点归类）
2. 不新增矩阵的列百分比模式（仅行百分比）
3. 不改变前后置关联分析的SQL逻辑（数据口径保持不变）
4. 不新增矩阵的导出功能
