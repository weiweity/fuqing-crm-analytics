## 1. Problem Statement

### 三个用户可见痛点（不是 bug，是架构问题）

| 痛点 | 表面现象 | 根因（SaaS 视角）|
|---|---|---|
| **P1 ETL 慢** | 单次 41-63min，运营等到冒烟 | **Compute layer 缺增量** — 全量重算 540 组合 × 全历史 |
| **P2 RFM 偶尔错** | 总人数突降、回购=0 | **没有 atomic snapshot** — ETL 写一半 API 读半新半旧；或 540 循环里 1 个失败被吞 |
| **P3 历史查询慢** | 选 2026-01-01→2026-01-31 重算 | **没有 precomputed serving layer** — 把 DuckDB 当 MySQL 用，每次 query 都现算 |

### 互相恶化的反身性

```
只解 P1 → 加速 ETL → 更多人想查历史 → P3 更严重
只解 P3 → 加预计算 → 新 ETL 步骤 → P1 更严重
只解 P2 → 修一致性 → 不解决任何 perf 问题
```

**必须三件套一起做（boil the lake）**，否则永远在修一个痛点的时候让另一个恶化。

### 选定的目标架构（SaaS 4 层标准）

```
┌─────────────────────────────────────────────────────────┐
│  Layer 4 — Query API                                    │
│  查 v_latest_rfm VIEW + DuckDB-KV 缓存 (TTL 24h)        │
│  选 2026-01-01→2026-01-31 → 1ms 命中 cache              │
└─────────────────────────────────────────────────────────┘
              ↑ 读
┌─────────────────────────────────────────────────────────┐
│  Layer 3 — Precomputed Serving Tables                   │
│  fact_rfm_long × 540 dims × 全历史日期（写死）          │
│  Z-ORDER BY (date, channel, item) → 切片查询 O(scan)    │
│  dbt-style snapshot：late-arriving merge 修复 T-7       │
└─────────────────────────────────────────────────────────┘
              ↑ 原子切换 (manifest.json)
┌─────────────────────────────────────────────────────────┐
│  Layer 2 — ETL Pipeline                                 │
│  Step 1-7: load → transform → upsert → status_override  │
│            → refresh_visitor → precompute_rfm           │
│  Step 8:  DQ assertions（修 W3）                        │
│  Step 9:  Atomic manifest swap（修 W2）                 │
│  540 循环 → GROUPING SETS 1 query（修 W1）              │
└─────────────────────────────────────────────────────────┘
              ↑ 流式
┌─────────────────────────────────────────────────────────┐
│  Layer 1 — Source Data                                  │
│  xlsx → Parquet 缓存 → DuckDB orders (10.6M)            │
└─────────────────────────────────────────────────────────┘
```

## 2. Premises（决策的事实基础）

| # | Premise | 验证方式 | 反驳条件 |
|---|---|---|---|
| 1 | DuckDB 41GB 单文件 OLAP，加 atomic view 切换是 SaaS 标准方案 | 业界 Looker/Tableau/Hex 通用 | 5 亿+ 行级别需换 ClickHouse |
| 2 | 10.6M 订单量级，DuckDB 表当 KV 缓存够用（< 100K 查询 hash） | 实测：10M 行级别 DuckDB random read < 10ms | 单日查询 > 1M 才需真 Redis |
| 3 | manifest.json 是"足够 atomic"的开关 | ETL + API 都在同一台机器（Mac/Windows 部署）| 多机部署需分布式锁 |
| 4 | 540 组合 GROUPING SETS 合并可砍 60-70% preload 时间 | baseline 已确认 540 loop 占 44.9% wall | 实测 GROUPING SETS 比 loop 慢则推翻 |
| 5 | 6+ DQ 断言能覆盖 95% 脏数据场景 | cross_day / api_health / copy_day 已有类似设计 | 加更多断言边际收益递减 |
| 6 | (date, version) 唯一索引实现幂等 | DuckDB 支持唯一约束 | ETL 跨日有重复订单需扩到 (order_id, version) |
| 7 | 纯增量 + dbt-style snapshot 适合 10.6M 订单 | Late-arriving 订单 T-7 内覆盖 99% | 跨月大单需扩到 T-30 |

**关键设计**：

```python
# scripts/etl/manifest.py
class SnapshotManifest:
    """atomic switch via single-file JSON write + rename"""
    def write_active(self, view_name: str) -> None:
        # Write to tmp + os.rename (atomic on POSIX)
        tmp = f"{self.path}.tmp.{os.getpid()}"
        with open(tmp, "w") as f:
            json.dump({"active_view": view_name, "version": self._next_version(), "ts": ...}, f)
        os.rename(tmp, self.path)  # atomic on POSIX, near-atomic on Windows
    
    def read_active(self) -> str:
        # API 层每次查询读这个
        with open(self.path) as f:
            return json.load(f)["active_view"]
```

**API 层集成**：
```python
# backend/services/rfm/loader.py
def get_rfm_view_name() -> str:
    return SnapshotManifest(MANIFEST_PATH).read_active()

# 然后 SELECT * FROM {{ get_rfm_view_name() }} WHERE ...
```

```python
def incremental_load(target_date: date) -> None:
    """T+1 时跑，只处理 T-1 当天数据 + merge T-7 修复 late-arriving"""
```

### W5: API 层 DuckDB-KV 缓存

**目标**：rfm_query_cache 表，TTL 24h，命中 < 5ms
**位置**：新文件 `backend/services/rfm/cache.py`
**估时**：CC 1d / 你 0.5d

**关键设计**：
```python
# backend/services/rfm/cache.py
class RfmQueryCache:
    def get(self, date_range: str, dims: list[str]) -> dict | None:
        key = self._hash(date_range, dims)
        row = self.conn.execute(
            "SELECT value FROM rfm_query_cache WHERE key=? AND expire_at > now()",
            [key]
        ).fetchone()
        return json.loads(row[0]) if row else None
    
    def set(self, date_range, dims, result) -> None:
        key = self._hash(date_range, dims)
        self.conn.execute(
            "INSERT OR REPLACE INTO rfm_query_cache VALUES (?, ?, ?)",
            [key, json.dumps(result), datetime.now() + timedelta(hours=24)]
        )
```

**API 层集成**：
```python
# backend/services/rfm/api.py
@router.get("/api/v1/rfm/summary")
def get_rfm_summary(date_from, date_to, dims):
    cached = cache.get(f"{date_from}_{date_to}", dims)
    if cached:
        return cached  # ~1ms
    result = compute_rfm_summary(date_from, date_to, dims)  # 走 fact_rfm_long
    cache.set(f"{date_from}_{date_to}", dims, result)
    return result
```

**manifest 变更触发整表失效**：
```python
# backend/main.py startup
def invalidate_cache_on_manifest_change():
    current = manifest.read_active()
    if current != self._last_seen_version:
        conn.execute("DELETE FROM rfm_query_cache")
        self._last_seen_version = current
```

- [ ] 4 个 RFM 端点都加 cache（summary / segment / flow / repurchase）
- [ ] manifest 变更检测 + 整表失效

**CLAUDE.md 合规**（v1.1 增）：
- ① **ThreadSafeCursor 包装**（reference.md 教训 2026-05-31：DuckDB execute 锁只覆盖执行期间，结果集绑定在连接上；多 API 线程并发 fetch 会读到对方结果）。`RfmQueryCache.get/set` 必须用 `ThreadSafeCursor` 在锁内预取全部结果到内存。

```python
# scripts/etl/notify.py
import os, subprocess, json
from datetime import datetime

def notify_etl_complete(stats: dict) -> tuple[bool, str]:
    """ETL 跑完通知。复用 6 道门禁的 lark-cli 通道（scraper/core/sanity_check.py:_send_lark_alert）"""
    oids = os.environ.get("NOTIFY_OPEN_IDS", "").split(",")
    oids = [o.strip() for o in oids if o.strip()]
    if not oids:
        return False, "未配置 NOTIFY_OPEN_IDS，跳过通知"
    
    msg = (
        f"✅ ETL 跑完 {datetime.now().strftime('%Y-%m-%d %H:%M')}\n"
        f"orders: {stats.get('orders_count', '?')}\n"
        f"user_rfm: {stats.get('user_rfm_count', '?')}\n"
        f"wall time: {stats.get('wall_min', '?')}min\n"
        f"6 道门禁: {stats.get('gates_overall', '?')}"
    )
    bin_path = os.environ.get("LARK_BIN", "/Users/hutou/homebrew/bin/lark-cli")
    results = []
    for oid in oids:
        proc = subprocess.run(
            [bin_path, "im", "+messages-send", "--user-id", oid, "--text", msg, "--as", "bot"],
            capture_output=True, text=True, timeout=5.0,
        )
        results.append(proc.returncode == 0)
    return all(results), f"{sum(results)}/{len(oids)} 推送成功"
```

## 8. Failure Modes（每条路径的现实风险）

| Codepath | 失败模式 | 是否有测试 | 错误处理 | 用户看到 |
|---|---|---|---|---|
| W1 GROUPING SETS | SQL 语法错（漏 GROUPING）| ✓ | pytest 必跑 | 540 跑空 → 500 |
| W2 manifest write | 写一半断电 | ✓ | atomic rename | 旧 manifest 仍可读 |
| W2 manifest read | manifest 文件丢失 | ✗ | 默认 fallback 到 latest_known_view | 200（降级）|
| W3 assertions | 断言全失败 | ✓ | 全部入 quarantine，ETL 继续 | 200 但数据 stale |
| W4 fact_rfm append | UNIQUE 冲突 | ✓ | ON CONFLICT DO NOTHING | 200 |
| W4 late-arriving | T-30 之后才到 | ✗ | 走 rfm_recompute_window 手动 | 200 但 stale |
| W5 cache stale | manifest 改了但 cache 没失效 | ✓ | version 校验 | 200 但数据旧 |
| W5 cache poisoning | hash 碰撞 | ✗ | 用 SHA-256 几乎不可能 | 极小概率 |

## 9. Risks & Mitigations

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| 全历史跑一次 30min+ 失败 | 中 | 中 | 分批跑（季度为单位），checkpoint 续传 |
| DuckDB 41GB 内存不足 | 低 | 高 | 跑 W4 时临时调 memory_limit=16GB |
| manifest 不 atomic on Windows | 中 | 中 | 加 fsync 强制刷盘；测试覆盖 Windows 部署 |
| 540 GROUPING SETS 实际比 loop 慢 | 低 | 中 | W1 先做 1 次 benchmark，慢了回滚 |
| cache 表无限增长 | 中 | 低 | 定期 cleanup（> 7d 删）|

## 16. CLAUDE.md 12 步 check-list per WO（v1.1 增）

每个 WO 必须走完 12 步流程，标准任务清单：

| 步 | 标准任务 | 适用 WO |
|---|---|---|
| ① | `git checkout -b {branch}`（命名规范见下）| 全部 |
| ② | 写代码（改 .py / 新文件）| 全部 |
| ③ | `PYTHONPATH="$(pwd)" pytest backend/tests/ -x -q`（必须用 homebrew Python 3.14）| 全部 |
| ④ | /review skill（自检 lint + ruff + 单测 + docstring）| 全部 |
| ⑤ | 修 review 找出的问题 | 全部 |
| ⑥ | `git commit -m "feat: ..."` 或 `fix: ...`（不用 "fix"/"update"）| 全部 |
| ⑦ | `git push origin {branch}` | 全部 |
| ⑧ | /qa skill（pytest + ruff + 签名校验 + E2E 关键场景）| 全部 |
| ⑨ | `git checkout main && git merge {branch} --no-ff` | 全部 |
| ⑩ | `git push origin main` | 全部 |
| ⑪ | `git pull origin main --ff-only` | 全部 |
| ⑫ | kill + restart uvicorn + 更新 CHANGELOG | 全部 |

**分支命名规范**（CLAUDE.md 强制）：
- W1 (bug fix): `fix/wo1-mt1-grouping-sets`
- W2-W7 (新功能): `feat/wo2-manifest-snapshot` / `feat/wo3-dq-assertions` / `feat/wo4-fact-rfm-long` / `feat/wo5-rfm-cache` / `feat/wo6-etl-notify` / `feat/wo7-memory-limit`

**commit message 规范**（CLAUDE.md 强制）：
- ✅ `feat: W2 原子 manifest.json 切换`
- ✅ `fix: preload_rfm.py 540 GROUPING SETS 重写`
- ❌ `fix` / `update` / `change`（无类型前缀）
- ❌ 一个 commit 混多个不相关功能

**每个 WO 的特殊 12 步附加项**：
- **W1**: qa 验 ETL wall < 70min（CLAUDE.md: 跑 ETL 验证）| 验 row count 1:1
