## 1. 定位与职责

服务层（Service Layer）是**业务逻辑的核心承重层**，位于语义层与 API 路由层之间。

**数据流向**

```
请求 → main.py 路由 → Service（业务逻辑）→ 语义层（口径定义）→ DuckDB
                      ↑
              Service 引用语义层获取口径，
              自身不定义任何 SQL 过滤条件
```

> ⚠️ **核心约束（强制）**
> - Service 中**禁止硬编码** `is_goujinjin = FALSE`、`is_refund = FALSE`、`order_status LIKE '%成功%'` 等口径逻辑
> - 所有口径统一从 `backend.semantic.*` 引入
> - Service 只负责：参数组装 → 调用语义层 → 结果转换 → 返回契约模型

## 4. 设计模式

### 4.1 标准 Service 结构 ✅

```python
from backend.semantic.filters import OrderFilters, AmountExprBuilder
from backend.contracts.schemas import XxxResponse

def calculate_xxx(params: XxxRequest) -> XxxResponse:
    # ① 参数验证
    date_range = build_date_range(params)

    # ② 口径构造（引用语义层，禁止硬编码）
    valid_filter = OrderFilters.valid_order()         # is_goujinjin=FALSE AND NOT is_refund
    amount_expr  = AmountExprBuilder.gmv()             # 剔除购物金的金额表达式

    # ③ SQL 执行
    sql = f"SELECT {amount_expr} AS gmv FROM orders WHERE {valid_filter} AND ..."
    result = duckdb.execute(sql)

    # ④ 结果转换 → 契约模型
    return XxxResponse(**result)
```

### 4.2 禁止的模式 ❌

```python
# ❌ 禁止：在 Service 中硬编码口径
def calculate_xxx(params):
    sql = """
        SELECT SUM(actual_amount) AS gmv
        FROM orders
        WHERE order_status NOT LIKE '%关闭%'   -- 硬编码口径 ❌
          AND order_status NOT LIKE '%退款%'   -- 硬编码口径 ❌
          AND is_goujinjin = FALSE             -- 硬编码口径 ❌
    """
```

## 5. 契约层集成

Service 返回的数据类型统一为 `contracts/schemas.py` 中定义的 Pydantic 模型：

```python
from backend.contracts.schemas import OverviewMetrics, DateRangeResponse

def calculate_metrics(...) -> OverviewMetrics:
    return OverviewMetrics(
        metric_type=metric_type,
        date_range=DateRangeResponse(...),
        amount=gmv,
        ...
    )
```

> 📌 **规则**：Service 的返回类型注解必须与 `contracts/schemas.py` 中的模型名称完全一致。
