📐 AI 问数 Pipeline 详解

① 解析层
输入: 用户问数
输出: 意图 + 实体
→
② 检索层
输入: 解析结果
输出: 相关表/字段
→
③ 生成层
输入: Schema + Prompt
输出: SQL
→
④ 校验层
输入: SQL
输出: 合规 SQL
→
⑤ 执行层
输入: 合规 SQL
输出: 结果集
→
⑥ 包装层
输入: 结果集
输出: NLG 描述
P0 = 立刻做（不补就盲飞）
P1 = 下一步（评估驱动优化）
P2 = 进阶（竞争壁垒）

⚠️ 你现在缺什么？（核心差距）

❌ 没有评估集
不知道 AI 生成的 SQL 对不对，无法量化改进效果
→ 收集 50-100 个标注好的 query-SQL 对
❌ 没有校验层
生成的 SQL 直接进数据库，可能有语法错误或恶意查询
→ DuckDB 原生校验 + 简单 regex 过滤
❌ 没有监控
哪些 query 回答得好/差？执行耗时多少？全靠感觉
→ 记录每次 query + SQL + 执行时间 + 结果质量
❌ 没有 Prompt 版本管理
Prompt 改来改去，不知道哪个版本更好
→ Prompt 模板库 + A/B 测试机制

P0 立刻补上

  • 建立 SQL 正确性评估集
    50-100 个历史 query，手工标注正确 SQL
  • 添加 SQL 语法校验层
    防止乱 SQL 进数据库 / DuckDB 原生校验
  • 添加结果 sanity check
    子字段和 ≤ 总数、比例合理、值域过滤

P1 评估驱动优化

  • Prompt 版本管理
    模板库 + 版本对比 + 回答质量评分
  • 慢查询分析
    统计 Top 10 慢查询 / SQL 执行时间分布
  • NLG 描述优化
    让 AI 返回自然语言解释，不只是数字

P2 竞争壁垒

  • Text-to-SQL 微调
    用你们数据库的 query-SQL 对微调模型
  • 历史 query 记忆
    同一个用户问类似问题时复用上下文
  • 多表关联优化
    跨品类、跨地域查询的 SQL 质量提升
