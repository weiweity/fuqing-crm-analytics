# 增量 ETL 失败排查报告

**日期**: 2026-04-17  
**问题**: 执行 `--update` 增量跑数时，店铺/会员数据均显示"无新增文件，跳过"，最终报错 `错误: 没有加载到任何店铺数据!`  
**状态**: 已修复并验证通过

---

## 一、现象回放

执行命令：
```bash
python scripts/run_etl.py --update
```

输出关键日志：
```
加载数据: .../店铺数据库 (模式: incremental)
  [Parquet 缓存] 无新增文件，跳过

加载数据: .../会员数据库 (模式: incremental)
  [Parquet 缓存] 无新增文件，跳过

错误: 没有加载到任何店铺数据!
```

---

## 二、根因分析

### 2.1 直接原因
`scripts/run_etl.py` 的 `load_data_files()` 函数在增量模式下存在**逻辑短路 bug**：

```python
# 原代码（bug 版本）
if run_mode == 'incremental':
    new_files = [f for f in pq_files if f.name not in processed_files]
    if not new_files:
        print("  [Parquet 缓存] 无新增文件，跳过")
        return pd.DataFrame()   # <-- 致命：直接返回空，跳过了 xlsx fallback
```

当 `data/parquet/shop/` 和 `data/parquet/member/` 目录存在且其中的 parquet 文件已全部被标记为"已处理"时，即使原始 `xlsx` 目录中有新增文件，函数也会直接返回空 DataFrame，根本不会走到下方的 **xlsx fallback** 逻辑。

### 2.2 为什么之前没发现
- 全量 ETL 时正常（走全量 parquet 读取或 xlsx 读取）。
- 之前增量跑数时，可能恰好有新增 parquet 文件（例如先执行过 parquet 转换脚本），因此 bug 未触发。
- 本次跑数前，parquet 缓存已与 processed_files 完全同步，而原始 xlsx 目录又新增了 2 个文件，导致 bug 暴露。

### 2.3 数据验证
| 数据源 | 总 xlsx 数 | processed 记录数 | 未处理 xlsx 数 |
|--------|-----------|------------------|----------------|
| 店铺数据库 | 61 | 120* | 1 |
| 会员数据库 | 36 | 70* | 1 |

\* processed 记录数偏高的原因：`processed_files_*.json` 中**混合存储**了 xlsx 相对路径和 parquet 文件名（不影响功能，但内容较乱）。

未处理的 2 个文件：
- `店铺数据库/26年数据库_260320182946/芙清旗舰店_数据营销_购买订单_活动1752_任务20772_095319848.xlsx`
- `会员数据库/26年会员数据库_260320182957/芙清旗舰店_数据营销_购买订单_活动1752_任务20773_095617949.xlsx`

## 三、修复方案

### 3.1 代码修复
**文件**: `scripts/run_etl.py`  
**修改点**: `load_data_files()` 函数（约第 409-453 行）

将原来的"无新增即 `return pd.DataFrame()`"改为"无新增时打印提示并继续往下走 xlsx fallback"。

修复后核心逻辑：
```python
if run_mode == 'incremental':
    new_files = [f for f in pq_files if f.name not in processed_files]
    if not new_files:
        print("  [Parquet 缓存] 无新增 parquet 文件，继续检查 xlsx 原始文件")
        should_read_parquet = False   # 跳过 parquet 读取块
    else:
        pq_files = new_files
        print(f"  [Parquet 缓存] 增量模式: {len(pq_files)} 个新增文件...")

if should_read_parquet:
    # 读取 parquet...
    if all_data:
        return combined
# 如果 parquet 没读到数据，继续走下面的 xlsx fallback
```

### 3.2 额外建议（非紧急）
- **processed_files 去混存**: 当前 JSON 中同时存在 xlsx 相对路径和 parquet 文件名。建议未来拆分两个 tracker（`processed_parquet_*.json` 和 `processed_xlsx_*.json`），避免维护混乱。

## 五、结论

- **根因**: 增量 ETL 的 `load_data_files()` 在 parquet 无新增时过早 `return pd.DataFrame()`，导致 xlsx 原始文件的新增数据被漏检。
- **影响**: 本次跑数前新增的 2 个 xlsx 文件（共 4,771 行店铺数据 + 2,020 行会员数据）未被导入。
- **修复**: 已移除过早返回，确保增量模式始终会检查 xlsx fallback。
- **验证**: 修复后增量跑数成功，新增数据已正确入库，数据验收门卫全部通过。
