# 达摩盘流转数据 - 新增人群Bug修复报告

**日期**: 2026-04-06
**问题**: 新增人群(xinzeng)流转数据全为0

---

## 问题现象

3月19日新增人群数据：
| 指标 | 错误值 | 正确值 |
|------|--------|--------|
| initial | 2,891,201 ✅ | 2,891,201 ✅ |
| →发现 | **0** ❌ | 791,168 ✅ |
| →种草 | **0** ❌ | 2,095,114 ✅ |
| →互动 | **0** ❌ | 772 ✅ |
| →行动 | **0** ❌ | 2,938 ✅ |
| →首购 | **0** ❌ | 1,207 ✅ |

---

## 做得好点

1. **日志完善**: 发现问题后，通过添加详细日志迅速定位到 transfer/channel API 在访问 statusId=0 时没有返回数据

2. **API方案正确**: 对 2001-2008 的7个阶段，API拦截方案完全正确，数据准确

3. **备用方案意识**: 早在设计时就预留了 DOM 提取作为备选方案

4. **DOM定位准确**: 快速定位到 left=1028 区域是新增流转数据的显示位置

5. **数据结构清晰**: 使用 top/20 取整成功去重，映射关系正确

---

## 做得坏点

1. **过度依赖API**: 盲目认为所有数据都能通过 transfer/channel API 获取，未预判到 statusId=0 的特殊性

2. **title匹配脆弱**: 当 title 不包含"新增"时直接丢弃数据，应该用 URL 中的 statusId 辅助判断

3. **只访问一次**: 循环中按顺序访问8个statusId，但 transfer/channel API 的返回时机不一致，导致 statusId=0 的数据被遗漏

4. **缺少边界测试**: 对 statusId=0 这个特殊值缺少测试用例

5. **修复方式绕弯**: 花了较多时间在"为什么API不返回"的排查上，应该更早改用DOM方案

---

## 解决方案

### 1. 主方案：API拦截 + URL参数化
```python
# 对 2001-2008 的7个阶段，通过 transfer/channel API 获取流转数据
for status_id in [2001, 2002, 2003, 2004, 2006, 2007, 2008]:
    url = f"{DMP_BASE_URL}?spm={spm}{route}?statusId={status_id}&..."
    page.goto(url)
    # 拦截 API 响应，提取 transforms[].uv
```

### 2. 特殊处理：statusId=0 用 DOM 提取
```python
# 新增函数：extract_xinzeng_flow_by_dom()
# 访问 statusId=0 页面后，从 DOM 提取 left=1028 区域的数字
# 按 top 位置映射：486→发现, 552→种草, 618→互动, 684→行动, 750→首购
```

### 3. 根本原因
transfer/channel API 对 statusId=0 **不返回**或**延迟返回**流转数据，可能原因：
- 新增是"流入"人群，不是"流出"人群
- 桑基图的源是7个阶段，不是8个
- 新增的流转数据需要特殊处理

### 4. 预防措施
- 对所有 statusId 都应该有数据来源的备选方案
- 写单元测试覆盖 statusId=0 的场景
- 日志应该打印 "未匹配title" 的实际值，便于排查
